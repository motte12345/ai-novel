## 2026-05-19: max_tokens は日本語の字数制限として効きにくい（Llama tokenizer の罠）

### 問題
Phase 1 PoC で「1章 300〜400字」を目指して `max_tokens=500` と設定したが、Groq Llama 4 Scout / Llama 3.1 8B の実出力が **411〜724字** に。max_tokens=400 に下げても 411〜609字。物理制限を強くしても字数は思ったほど減らない。

### 原因
**Llama 系 (3.x / 4) の tokenizer は日本語の頻出語を 1 token に統合する**。
- 「彼女は」「自然」「である」「と思った」のようなフレーズが 1 〜 2 token
- 結果 1 文字 ≒ 0.5 〜 0.8 token 程度の効率になり、500 token = 700+ 文字も書ける
- 一方 Gemini の tokenizer は 1 文字 ≒ 1 token に近い

ai-roundtable で「150〜250字 / max_tokens 350」が機能していたのは、150-250字の**プロンプト指示自体が短文寄りに引き寄せた**効果と、Llama 3.3 70B が指示追従能力で多少守った効果の合成。300-400字を狙うとプロンプトの引力が弱くなり物理制限も効かないので破綻する。

### 解決策（PoC 3回目で確立）
1. **目標字数を 200〜300字に下げる**: プロンプトと max_tokens 双方で短文に誘導
2. **max_tokens=300（作家）**: トータルで多少のオーバーは許容
3. **`trimToLastSentence()` でポストプロセス**: 出力の末尾を句点単位で切る。max_tokens 途中切れと字数超過の両方に対応
4. **プロンプトの強化**: 「200〜300字を絶対厳守。1字でも350字を超えたら不採用」「冗長な説明・言い換えで字数を稼がない」と明示

### 教訓
- **Llama 系の max_tokens は字数制限として直感的に効かない**。日本語想定なら token ÷ 0.7 くらいの文字数になる
- 字数を厳しく守らせたいなら、目標を**短めに振る + プロンプト強化 + 末尾トリム** の3層
- ai-roundtable と ai-novel の経験から、Llama × 日本語短文制約は **「~指示字数の1.3倍程度に膨らむ前提」** で設計すべし

## 2026-05-19: Llama 3.1 8B は結末担当には品質不足（ループ・破綻）

### 問題
PoC 1回目で writer_c の primary を Llama 3.1 8B にしたところ、最終章（第6章）が:
- 「町に戻る → 新しく始めた → また自然へ → 心の余白」と**同じ要素のループ**
- 文末がぶつ切れ「彼女は、自然の中で歩く」
- 過去章の言い回しを意味も変えず反復する
- 教訓化（「〜が大切だと感じた」）に流れる

### 原因
Llama 3.1 8B はパラメータ数が小さく、**多文脈の統合・象徴の選択・余韻のある締めくくり**といった高次の文芸的判断が弱い。短い指示への即応や軽い対話には適するが、6章分の文脈を踏まえた結末は荷が重い。

### 解決策
- **writer_c の primary を Groq Llama 4 Scout に格上げ**
- 8B は fallback 専用に降格
- 結末担当（最終章）のプロンプトに「新キャラ・新場面禁止」「既出の象徴を1つ拾って閉じる」「教訓化禁止」を明示

PoC 3回目で writer_c が Scout になり、最終章が「彼女は静かに窓を閉じ、部屋の薄いカーテンが風に揺れるのを感じながら、じっと立ち尽くした」のように、雀（既出の象徴）を心の中の存在に変換しつつ余韻で閉じられた。

### 教訓
- **モデル選定は役割の難易度で決める**。短い対話や定型処理なら 8B で十分、創作の結末のような統合判断には Scout 以上を割く
- ai-roundtable の Skeptic で Scout が活躍した実績は、似た「論点の統合と一刺し」を要求される役割にも当てはまる

## 2026-05-19: editor を Gemini Flash-Lite → Groq Scout に振り替え

### 問題
editor の primary に Gemini Flash-Lite を割り当てていたが、PoC 2回目で **HTTP 429 (Quota exceeded)** が editor ターンで発生。ai-roundtable と Gemini RPD を取り合っているため。

### 解決策
- editor の primary を **Groq Llama 4 Scout** に変更（PoC 3回目で実証、構造化フォーマット【ペンネーム】【作品タイトル】【章タイトル】も問題なく出力）
- Gemini Flash-Lite は fallback 専用に降格
- ai-novel から Gemini 系を呼ぶのは writer_a のみに

### 教訓
- 構造化出力は Scout でも十分こなせる。Gemini を選ぶのは「自然な短文・低レイテンシ」が欲しいときだけ
- 同一アカウントで複数サイトを動かす場合、**プロバイダの主軸を分散させる**設計が効く（ai-roundtable=Gemini Flash+Groq 70B、ai-novel=Groq Scout+Llama 8B）
- PoC 段階で「他サイトとの合算消費」を観測しないと本番で気付かない
