## 2026-05-19: max_tokens は日本語の字数制限として効きにくい（Llama tokenizer の罠）

### 問題
Phase 1 PoC で「1章 300〜400字」を目指して `max_tokens=500` と設定したが、Groq Llama 4 Scout / Llama 3.1 8B の実出力が **411〜724字** に。max_tokens=400 に下げても 411〜609字。物理制限を強くしても字数は思ったほど減らない。

### 原因
**Llama 系 (3.x / 4) の tokenizer は日本語の頻出語を 1 token に統合する**。
- 「彼女は」「自然」「である」「と思った」のようなフレーズが 1 〜 2 token
- 結果 1 文字 ≒ 0.5 〜 0.8 token 程度の効率になり、500 token = 700+ 文字も書ける
- 一方 Gemini の tokenizer は 1 文字 ≒ 1 token に近い

ai-roundtable で「150〜250字 / max_tokens 350」が機能していたのは、150-250字の**プロンプト指示自体が短文寄りに引き寄せた**効果と、Llama 3.3 70B が指示追従能力で多少守った効果の合成。300-400字を狙うとプロンプトの引力が弱くなり物理制限も効かないので破綻する。

### 解決策（PoC 3回目で確立）
1. **目標字数を 200〜300字に下げる**: プロンプトと max_tokens 双方で短文に誘導
2. **max_tokens=300（作家）**: トータルで多少のオーバーは許容
3. **`trimToLastSentence()` でポストプロセス**: 出力の末尾を句点単位で切る。max_tokens 途中切れと字数超過の両方に対応
4. **プロンプトの強化**: 「200〜300字を絶対厳守。1字でも350字を超えたら不採用」「冗長な説明・言い換えで字数を稼がない」と明示

### 教訓
- **Llama 系の max_tokens は字数制限として直感的に効かない**。日本語想定なら token ÷ 0.7 くらいの文字数になる
- 字数を厳しく守らせたいなら、目標を**短めに振る + プロンプト強化 + 末尾トリム** の3層
- ai-roundtable と ai-novel の経験から、Llama × 日本語短文制約は **「~指示字数の1.3倍程度に膨らむ前提」** で設計すべし

## 2026-05-19: Llama 3.1 8B は結末担当には品質不足（ループ・破綻）

### 問題
PoC 1回目で writer_c の primary を Llama 3.1 8B にしたところ、最終章（第6章）が:
- 「町に戻る → 新しく始めた → また自然へ → 心の余白」と**同じ要素のループ**
- 文末がぶつ切れ「彼女は、自然の中で歩く」
- 過去章の言い回しを意味も変えず反復する
- 教訓化（「〜が大切だと感じた」）に流れる

### 原因
Llama 3.1 8B はパラメータ数が小さく、**多文脈の統合・象徴の選択・余韻のある締めくくり**といった高次の文芸的判断が弱い。短い指示への即応や軽い対話には適するが、6章分の文脈を踏まえた結末は荷が重い。

### 解決策
- **writer_c の primary を Groq Llama 4 Scout に格上げ**
- 8B は fallback 専用に降格
- 結末担当（最終章）のプロンプトに「新キャラ・新場面禁止」「既出の象徴を1つ拾って閉じる」「教訓化禁止」を明示

PoC 3回目で writer_c が Scout になり、最終章が「彼女は静かに窓を閉じ、部屋の薄いカーテンが風に揺れるのを感じながら、じっと立ち尽くした」のように、雀（既出の象徴）を心の中の存在に変換しつつ余韻で閉じられた。

### 教訓
- **モデル選定は役割の難易度で決める**。短い対話や定型処理なら 8B で十分、創作の結末のような統合判断には Scout 以上を割く
- ai-roundtable の Skeptic で Scout が活躍した実績は、似た「論点の統合と一刺し」を要求される役割にも当てはまる

## 2026-05-21: 「穏当な決着」に逃げる LLM の根本傾向と対策

### 問題
プロンプトに「台詞を入れる」「複数人物を出す」「起承転結」を入れた後でも、物語が「**何も起きずに皆を見守って終わる**」傾向が続いた。
- 例『安らぎの椅子』: 姉→弟→母 と人物は登場するが、誰も決断せず、何も明らかにならず、「静かな時間を共有する」で閉じる
- 「絆を感じる」「温もりに包まれる」「優しく見守る」のような**穏当な決着フレーズ**で物語を畳んでしまう

### 原因（複合）
1. **プロンプト自体に穏当化バイアスを書き込んでいた**: 「大袈裟な事件は避ける」「日常の小さな機微」「静かな変化で十分」 — これらが LLM の保守的傾向を**むしろ強化**していた
2. **作家3人の役割が「機能分担」止まり**: 叙景/心理/対話 と分担しても、皆同じ「優しい目線」で書いてしまう。**価値観・好み・嫌うもの**が無いから無難に流れる
3. **章ステージの転換指示が弱い**: 「静かな変化で十分」と書いてしまったため、Ch4-5 の転換でも何も起きない
4. **temperature 0.9** はやや低めで、表現の振れ幅が出ない

### 解決策（4層で対処）
1. **穏当化指示を削除**: 「大袈裟な事件は避ける」「日常の小さな機微」「静かな変化で十分」を全て削る
2. **明示的に葛藤・秘密を要求**: 「登場人物には必ず**裏の動機**や**抱えるもの**を持たせる」「読者の予想を1回は裏切る」
3. **禁止フレーズの追加**: 「絆を感じる」「温もりに包まれる」「優しく見守る」「静かな時間を共有」 など穏当な決着の典型を**明示禁止**
4. **作家に価値観を付与**:
   - A: 「光景に翳りを見る」(明るい場面でも陰を、暖かい場面でも冷たさを)
   - B: 「人物の抱えるものを暴く」(優しさの裏の計算、平穏の裏の不安)
   - C: 「沈黙の重みを信じる」「物語に決定的な動きを与えることを恐れない」
5. **Ch4-5 で「告白/決断/発見/喪失/再会」のいずれかを必須化**
6. **temperature 0.9 → 1.0**: 振れ幅を最大に

### 効果
PoC『いとなみの便箋』で、妻が夫に隠して手紙を書く葛藤劇が初めて生まれた。Ch4「あの便箋が、ただの紙切れではないことを、誰よりも自分が知っている」「やっぱり、書くわ」という決断、Ch6「便箋がゆっくりと滑り落ち」「どうして、こんな手紙を書いたの?」という問いで終わる構成。

### 教訓（ai-roundtable と共通の root cause）
- **LLM はデフォルトで穏当・両論併記・無難な決着に流れる**。これは ai-roundtable の Skeptic / Optimist プロンプトでも同じ問題に当たり、明示禁止で解決した
- **「~を避ける」「~は不要」のような穏当化指示はバイアスを強化する**。書かない方がマシ、むしろ反対方向（葛藤・秘密を要求）に明示すべき
- **「機能分担」と「価値観分担」は別物**。役割（叙景/心理/対話）だけ分けても全員同じ目線で書いてしまう。「何を見つけるか」「何を信じるか」までキャラごとに分けないと差が出ない
- **temperature 1.0 は実用的に有効**。0.9 までは表現が穏当に丸まり、1.0 で振れ幅が出る

## 2026-05-21: ジャンル + 方向性タグ（タイトルと独立抽選）の効果

### 背景（仕様）
タイトル（単語ペア）と独立に、6軸（genre / tone / aftertaste / plot_arc / theme / atmosphere）をランダム抽選してプロンプトに渡す。意図は「SF × 牧歌的 × 復讐」のような**意図的な不協和**で予想外の物語を生むこと。

### 観察された効果
- 親和的な組み合わせ（恋愛 × 謎解き × 成長 など）は素直に物語の骨格として機能する
- 矛盾度の高い組み合わせは LLM が**片方に寄せる傾向**がある
  - 例: 「幻想 / コメディ」→ ほぼシリアスサスペンスに寄った、コメディ要素は消えた
  - これは LLM が「整合性を優先して片方を諦める」癖、と解釈
- 親和的な2値（例: ジャンル「群像劇 / 寓話」、雰囲気「幻想的 / 閉塞的」）は両方反映されやすい

### 仕様判断
- 矛盾組み合わせで片方が消えるのは**仕様として受容**
- 「両方を共存させよ」とプロンプトで明示しても完全制御は無理
- ただし**矛盾度の低い組み合わせ**はちゃんと両方効くので、複数指定は十分に価値ある（25% 確率で 2 個目追加）

### 教訓
- **タグの組み合わせ抽選は単一抽選より物語の多様性を増やす**（実証）
- **複数値の両立は LLM 任せにせず、許容範囲を広く取る**（強要するとぎこちなくなる）
- 矛盾組み合わせを許す代わりに、「同軸内で逆の値が同時に出る確率は低めに」設計するのも一案（現状は完全独立なのでたまに「明るい / 暗い」が出る、これは妙味として残す）

## 2026-05-21: D1 でカンマ区切り文字列 + LIKE 検索パターン

### 課題
タグの複数値（例: tone = "明るい,静か"）を SQLite で保存し、検索したい。

### 採用したパターン
- DB はカンマ区切り TEXT（`tone TEXT` カラムに `"明るい,静か"`）
- 検索は `WHERE ',' || tone || ',' LIKE '%,検索値,%'`
  - 前後にカンマを付けることで部分一致のミスマッチ（「失恋愛」が「恋愛」にヒット等）を防ぐ
  - インデックスは効かないが、stories テーブルは数千件規模を想定すれば許容

### 教訓
- **SQLite で配列 / 多値カラムを扱う場合の軽量解**: カンマ区切り + 境界文字付き LIKE
- 件数が膨大になったら別テーブル正規化（`story_tags(story_id, axis, value)`）に切り替える。インデックス効かせたいなら JSON1 拡張で `json_each` も候補
- API 側で受けるクエリは厳密に値リストで検証する（注入対策、ai-novel は事前に語彙が固定なのでリスト照合可能）

## 2026-05-23: 動的 OGP + RSS + sitemap の Worker 実装パターン

### 構成（ai-roundtable から踏襲、ai-novel で実装）
- `/sitemap.xml`: 完結作品の URL + lastmod を XML で返す（max-age=3600）
- `/rss.xml` `/feed.xml`: RSS 2.0、最新20件、各 item に title/link/pubDate/author
- `/story/:id`: bot UA を `lib/bot-ua.ts` で判定し、bot なら `api/og-html.ts` が OGP メタタグ入り HTML を返す。ブラウザには通常 SPA を返す

### bot UA 判定で気をつけたこと
- Twitterbot / facebookexternalhit / LinkedInBot / Slackbot / Discordbot / Googlebot 等の標準パターンを正規表現リストで管理
- 国内 SNS（Mastodon / Misskey / Bluesky）も含める
- 一般ユーザー（Chrome / Safari）には影響しないので副作用低い

### 教訓
- Cloudflare Workers Assets では SPA fallback（wrangler.toml の `not_found_handling = "single-page-application"`）が `/api/*` 以外を index.html に向ける。bot 用 OGP HTML を返したいパスは **fetch() で先に拾う**必要がある
- `_redirects` ファイルで `/* /index.html 200` を書くと Workers Assets が **無限ループ検出**でエラーを返す。SPA fallback は wrangler.toml だけで済ませる
- `tsc -b` が生成する `*.js` 中間ファイルは git 化の前に noEmit にして掃除する。さもないとリポジトリに大量に混入する
